# Telegram to Matrix Backfill

[Leer en español](README.es.md)

Import Telegram Desktop JSON exports into Matrix/Synapse while preserving the
original sender and timestamp of each message. The tool keeps local progress in
SQLite so reruns are idempotent for already sent events.

## What This Project Does

- Parses Telegram Desktop exports (`result.json`).
- Creates or reuses Matrix rooms per Telegram chat.
- Creates or updates local Matrix users for Telegram senders.
- Sends messages through a Matrix Application Service token using the original
  sender MXID and timestamp.
- Stores per-message progress in a local SQLite checkpoint database.

## Current Scope

Supported Telegram chat types:

- `private_group`
- `private_supergroup`
- `personal_chat`
- `saved_messages`

Ignored chat types:

- `broadcast_channel`
- `public_channel`
- `channel`

## Requirements

- Python `3.10+`
- `ffmpeg` available in `PATH`
- A Synapse homeserver
- A working Matrix Application Service registration
- A Synapse admin token

The command line expects:

- `--as-token`: Application Service token used to call Matrix client/media APIs.
- `--hs-token`: currently accepted for compatibility, but not used by the code.
- `--synapse-admin-token`: required to create/update local users and run admin
  room operations.

## Synapse / AppService Prerequisites

This project does not implement an Application Service server. It assumes you
already have an AppService registration and only reuses its token to impersonate
local users through `user_id=...` requests.

Official Synapse documentation:

- Latest Element Synapse docs:
  `https://element-hq.github.io/synapse/latest/application_services.html`
- Matrix.org Foundation docs for older Synapse branches:
  `https://matrix-org.github.io/synapse/latest/application_services.html`
- Synapse Admin API user docs:
  `https://matrix-org.github.io/synapse/latest/admin_api/user_admin_api.html`

At minimum, Synapse needs an Application Service registration file and that file
must be listed in `app_service_config_files` inside `homeserver.yaml`.

Minimal example:

```yaml
id: telegram-backfill
url: http://127.0.0.1:9999
as_token: CHANGE_ME_AS_TOKEN
hs_token: CHANGE_ME_HS_TOKEN
sender_localpart: telegram-backfill-bot
namespaces:
  users:
    - exclusive: true
      regex: '@tg_.*:example.com'
    - exclusive: true
      regex: '@telegram-backfill-bot:example.com'
  aliases: []
  rooms: []
```

And in `homeserver.yaml`:

```yaml
app_service_config_files:
  - /etc/matrix-synapse/appservice-telegram-backfill.yaml
```

Important notes:

- `--domain` must match the local Synapse `server_name`, not the homeserver URL.
- The generated MXIDs must belong to a namespace reserved by the AppService.
- `--bot-mxid` should be the AppService sender user or another local user inside
  the AppService namespace.
- The code creates local users via Synapse Admin API
  `PUT /_synapse/admin/v2/users/<user_id>`, so remote/federated MXIDs will not
  work as generated users.

## Configuration Files

Create local working copies from the canonical templates:

```bash
cp templates/user_map.yaml ./user_map.yaml
cp templates/room_map.yaml ./room_map.yaml
```

The repository intentionally keeps only the files under `templates/`. Your
working `user_map.yaml` and `room_map.yaml` should be local files created from
those templates.

### `user_map.yaml`

Manual Telegram-user-to-Matrix-user mapping.

Example:

```yaml
"123456789":
  mxid: "@alice:example.com"
  displayname: "Alice Example"
```

How it works:

- Key: Telegram user ID from the export (`from_id` / `actor_id`).
- `mxid`: Matrix user to use for that Telegram sender.
- `displayname`: display name to set when the tool ensures the Matrix user.

If a Telegram user is not present in `user_map.yaml`, the tool generates a local
MXID with this pattern:

```text
@tg_<normalized_display_name>_<telegram_user_id>:<domain>
```

Example:

```text
@tg_alice_example_123456789:example.com
```

Important behavior:

- Generated fallback mappings are deterministic for the current display name and
  Telegram ID.
- `user_map.yaml` is not auto-saved by the importer. It stays manual on purpose.
- If a Telegram user's display name changes across exports and you rely on
  autogenerated MXIDs, you may end up with a different generated MXID. For
  stable identities, add explicit entries to `user_map.yaml`.

### `room_map.yaml`

Telegram-chat-to-Matrix-room mapping.

Example:

```yaml
"987654321": "!abcdefg:example.com"
```

How it works:

- Key: Telegram chat ID from the export.
- Value: existing Matrix room ID to reuse.
- If a chat ID is missing, the tool creates a new private Matrix room.
- Successful room creations are persisted back into `room_map.yaml`.
- If a mapped room no longer exists on Synapse, the mapping is dropped and the
  room is recreated.

## How Users and Rooms Are Created

### User creation flow

For every Telegram sender found in the export:

1. Resolve the target MXID from `user_map.yaml`, or generate one.
2. Call Synapse Admin API to create or update that local user.
3. Set the Matrix display name from the configured or Telegram display name.
4. Join that user to the target room before sending their messages.

The first author seen in each chat becomes the initial room creator/admin from
the tool's point of view.

### Room creation flow

For every Telegram chat:

1. Check `room_map.yaml`.
2. If no mapping exists, create a private Matrix room named after the Telegram
   chat title.
3. Ensure `--bot-mxid` is joined.
4. Ensure the chat creator user is joined.
5. Promote the creator user to room admin.
6. Set room history visibility to `shared`.
7. Join the rest of the Telegram senders found in that chat.

## End-to-End Backfill Flow

For each supported Telegram message:

1. Parse chat metadata, sender, timestamp, text and optional media.
2. Group messages by Telegram chat.
3. Resolve the target Matrix room.
4. Resolve the sender MXID.
5. Prepare media if needed.
6. Send a Matrix event using:
   - `user_id=<sender_mxid>`
   - `ts=<original_telegram_timestamp_ms>`
7. Store the result in the SQLite checkpoint DB.

The checkpoint table is keyed by `(telegram_chat_id, telegram_message_id)`.
Messages already marked as `sent` are skipped on rerun.

## Media and Files

Actual current behavior:

- Images are converted to JPEG.
- Audio is reused as-is for `.ogg`, otherwise converted to OGG/Opus with
  `ffmpeg`.
- Videos are converted to MP4/H.264/AAC with `ffmpeg`.
- Generic files/documents are uploaded as `m.file` with the original filename
  and guessed MIME type.
- Temporary converted files are written to a temporary directory and removed
  automatically when the process exits.

Important limitations:

- If Telegram exported a placeholder like
  `(File not included. Change data exporting settings to download.)`, the tool
  cannot upload that file and the message ends up without media.
- Messages with media or documents send only the media/file payload. Captions or
  extra text attached to the same Telegram message are currently not emitted as
  a second Matrix message.
- Documents/files are not transformed beyond MIME detection. There is no special
  preview, thumbnail or richer handling beyond `m.file`.

## Known Issues

- Video conversion is currently unreliable in real use. The code tries to
  transcode videos to MP4/H.264/AAC through `ffmpeg`, but when conversion
  fails the message is marked as `skipped` and is not sent.
- If you already know video conversion is not useful in your dataset, run with
  `--skip-videos` so video messages are skipped immediately.
- Generic files/documents are supported only as plain `m.file` uploads. There
  is no richer handling such as previews, thumbnails, or document-specific
  metadata extraction.
- If Telegram did not export the underlying file and the JSON contains the
  `(File not included...)` placeholder, this tool cannot recover it. The
  message is processed without the missing attachment.
- Media captions are not preserved when the same Telegram message contains both
  text and an attachment. The current implementation sends only the media/file
  event payload.
- Stickers are detected by the parser but are not supported by the sender
  pipeline, so they are skipped as unsupported media.
- `user_map.yaml` is not auto-persisted, so stable user identities should be
  curated manually if you do not want generated MXIDs to depend on Telegram
  display names.

## Usage

### Dry run

Use this first to validate parsing and configuration:

```bash
python -m telegram_to_matrix.cli \
  --telegram-export /path/to/export/result.json \
  --media-dir /path/to/export \
  --homeserver https://matrix.example.com \
  --domain example.com \
  --as-token CHANGE_ME_AS_TOKEN \
  --hs-token CHANGE_ME_HS_TOKEN \
  --synapse-admin-token CHANGE_ME_ADMIN_TOKEN \
  --bot-mxid @telegram-backfill-bot:example.com \
  --state-db ./checkpoint.sqlite \
  --room-map ./room_map.yaml \
  --user-map ./user_map.yaml \
  --dry-run \
  --verbose
```

### Real import

```bash
python -m telegram_to_matrix.cli \
  --telegram-export /path/to/export/result.json \
  --media-dir /path/to/export \
  --homeserver https://matrix.example.com \
  --domain example.com \
  --as-token CHANGE_ME_AS_TOKEN \
  --hs-token CHANGE_ME_HS_TOKEN \
  --synapse-admin-token CHANGE_ME_ADMIN_TOKEN \
  --bot-mxid @telegram-backfill-bot:example.com \
  --state-db ./checkpoint.sqlite \
  --room-map ./room_map.yaml \
  --user-map ./user_map.yaml \
  --verbose
```

### Important flags

- `--dry-run`: parse and inspect messages without sending events.
- `--bot-mxid`: bot/user that acts when creating rooms and inviting members.
- `--since-message-id N`: start from a specific Telegram message ID.
- `--max-messages N`: limit total processed messages.
- `--retry-max-attempts N`: retry limit for transient HTTP/network failures.
- `--retry-base-delay S`: exponential backoff base delay.
- `--fail-fast`: stop on the first non-recoverable failure.
- `--skip-videos`: skip video messages without attempting conversion.
- `--verbose`: enable detailed logs.

## Maintenance Utilities

### Purge partially imported chats from the checkpoint

List current checkpoint stats:

```bash
python -m telegram_to_matrix.purge --state-db ./checkpoint.sqlite --list
```

Preview auto-selected partial chats (`sent > 0` and `failed > 0`):

```bash
python -m telegram_to_matrix.purge --state-db ./checkpoint.sqlite --auto-partial
```

Apply the purge and also remove matching room mappings:

```bash
python -m telegram_to_matrix.purge \
  --state-db ./checkpoint.sqlite \
  --room-map ./room_map.yaml \
  --auto-partial \
  --remove-room-map \
  --apply
```

### Delete Matrix rooms for a clean reimport

Preview one room reset:

```bash
python -m telegram_to_matrix.reset_matrix_rooms \
  --homeserver https://matrix.example.com \
  --synapse-admin-token CHANGE_ME_ADMIN_TOKEN \
  --room-map ./room_map.yaml \
  --state-db ./checkpoint.sqlite \
  --chat-id 987654321 \
  --dry-run
```

Apply the deletion and wait for completion:

```bash
python -m telegram_to_matrix.reset_matrix_rooms \
  --homeserver https://matrix.example.com \
  --synapse-admin-token CHANGE_ME_ADMIN_TOKEN \
  --room-map ./room_map.yaml \
  --state-db ./checkpoint.sqlite \
  --chat-id 987654321 \
  --wait-delete \
  --apply
```

Also remove generated `@tg_...` users created by previous imports:

```bash
python -m telegram_to_matrix.reset_matrix_rooms \
  --homeserver https://matrix.example.com \
  --synapse-admin-token CHANGE_ME_ADMIN_TOKEN \
  --room-map ./room_map.yaml \
  --state-db ./checkpoint.sqlite \
  --chat-id 987654321 \
  --wait-delete \
  --cleanup-tg-users \
  --apply
```
